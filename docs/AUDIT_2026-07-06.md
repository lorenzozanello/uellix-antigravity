## BLOCKER — Architecture

### [BLOCKER] Missing Row-Level Security (RLS)

**Severity:** CRITICAL  
**Category:** Access Control / Data Confidentiality  
**CWE:** CWE-284 (Improper Access Control)

**What it is:**
- Database has NO per-row access policies. All organization scoping is **application logic only**.
- A bug in any service (e.g., forgetting `eq(projects.organizationId, ctx.org.id)`) silently leaks data across orgs.
- Supabase service role key (exposed in `.env.local`) bypasses even application checks.

**Impact:**
- **HIGH:** Competitor organizations' data (projects, investments, outcomes, financial proxies) exposed
- Investor information, beneficiary data, financial calculations accessible to unauthorized orgs
- Compliance violation (data isolation requirement)

**Root cause:**
- Supabase RLS not enabled on any table
- Application relies on developer discipline to check `organizationId` in every query

**Recommended fix:**
1. Enable RLS on all tables: `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`
2. Create policies per table scoped to organization membership:
   ```sql
   -- Example: projects table
   CREATE POLICY projects_org_isolation ON projects
     FOR SELECT USING (organization_id IN (
       SELECT organization_id FROM organization_members 
       WHERE user_id = auth.uid() AND status = 'active'
     ))
   ```
3. Keep app-level checks as defense-in-depth
4. Timef estimate: 1-2 weeks (implementation + testing)

**Status:** ⚠️ NOT IMPLEMENTED

## HIGH — Security & Correctness

### [HIGH] Two Divergent Allocation Services (Correctness Hazard)

**Severity:** HIGH  
**Category:** Data Integrity / Business Logic  
**Related to:** Code Quality C1

**What it is:**
- `lib/pipeline/allocations.ts` (soft archive: `status = 'archived'`)
- `lib/pipeline/outcome-funder-allocations.ts` (hard delete: `DELETE FROM ...`)
- Both write `outcome_funder_allocations` table
- Wired via different server actions, different UIs

**Impact:**
- Row created in one UI can be deleted in another with different semantics
- Audit trail divergence: soft archive preserves history, hard delete loses it
- Hard delete violates append-only audit model (pipeline-hardening design)
- Cap validation uses different logic (Decimal.js vs parseFloat)

**Recommended fix (1 week):**
1. Choose ONE backend: keep `allocations.ts` (matches append-only model)
2. Migrate `allocation-form` components + `app/actions/allocations.actions.ts` to use `allocations.ts`
3. Delete `outcome-funder-allocations.ts` entirely
4. Consolidate server actions

**Status:** ⚠️ BLOCKS PRODUCTION HANDOFF

### [HIGH] Race Condition in Allocation Cap Check

**Severity:** HIGH  
**Category:** Concurrency / Data Integrity

**Location:** `lib/pipeline/outcome-funder-allocations.ts:118-136` (create), `:217-233` (update)

**What it is:**
- `calculateExistingAllocationTotal()` reads total
- Check `if (newTotal > 100) throw`
- Then insert
- **No serialization between check and write**
- Two concurrent requests can both pass check, both insert, result: >100% allocated

**Impact:**
- Over-allocation can occur silently
- Detected only at readiness check time (after calculation is attempted)
- SROI results become incorrect if funder breakdown math assumes allocations sum to ≤100%

**Recommended fix:**
1. Wrap read+insert in database transaction
2. Add `SELECT ... FOR UPDATE` on outcome's allocation rows
3. Or add DB-level constraint/exclusion (check constraint on outcome level)
4. Route all allocation writes through one consolidated service

**Status:** ⚠️ BLOCKS PRODUCTION HANDOFF

### [HIGH] No CSRF Protection on Server Actions

**Severity:** HIGH  
**Category:** Security / API Protection  
**CWE:** CWE-352 (Cross-Site Request Forgery)

**What it is:**
- 48+ Server Actions in admin, allocation, investment paths
- No CSRF token validation in any form submission
- Vulnerable to cross-origin requests from malicious website

**Attack example:**
- Attacker tricks user into visiting malicious.com
- Malicious.com makes cross-origin POST to `uellix.vercel.app/admin/organizations/setStatus`
- Request modifies org status without user consent

**Impact:**
- Admin settings can be modified from other sites
- Org status changed, quotas modified, access revoked

**Recommended fix:**
1. Generate CSRF token at request time: `crypto.randomUUID()`
2. Include in form hidden field
3. Validate on submission in each server action
4. Verify `SameSite=Lax` or `SameSite=Strict` on cookies (Next.js does this by default)

**Status:** ⚠️ IMPLEMENT THIS SPRINT

### [HIGH] Project Narrative May Contain PII Sent to Gemini

**Severity:** HIGH  
**Category:** Data Privacy / AI Integration  
**CWE:** CWE-200 (Exposure of Sensitive Information)

**What it is:**
- `narrativeText` and `theoryOfChangeSummary` are user-supplied
- Can contain beneficiary names, emails, phone numbers
- Sent to Gemini API (external service)
- If Gemini API logs, data is exposed

**Impact:**
- PII leakage to third-party AI provider
- Compliance violation (GDPR, CCPA, local privacy laws)

**Recommended fix:**

2. Enhance `sanitizeNarrative()` to redact:

3. Add `pii_present` flag to `impactNarratives` table
4. Block Stella usage if PII detected
5. Audit Gemini logs to ensure request/response logging disabled

### [HIGH] No Rate Limiting on File Uploads

**Severity:** HIGH  
**Category:** Denial of Service / Resource Exhaustion  
**CWE:** CWE-770 (Allocation of Resources Without Limits)

**What it is:**
- `createFileEvidenceForProject()` validates file size (max 25 MB)
- But no request rate limit
- Attacker can upload 10,000 files in quick succession → exhaust storage quota

**Impact:**
- Storage quota exhaustion
- Database bloat with orphaned evidence rows
- Supabase S3 cost explosion

**Recommended fix:**
1. Token bucket per user+project: max 10 uploads/hour
2. Implement via in-memory Map + middleware
3. Return 429 (Too Many Requests) if limit exceeded
4. Log attempts to audit logs

**Status:** ⚠️ IMPLEMENT THIS SPRINT

## MEDIUM — Quality & Performance

### [MEDIUM] Allocation Validation Uses Floating-Point Math

**Severity:** MEDIUM  
**Category:** Precision Error  
**CWE:** CWE-1025 (Precision Error)

**Location:** `lib/pipeline/outcome-funder-allocations.ts:94-95, 179-180`

**What it is:**
- Sums allocations using `parseFloat()` instead of `Decimal.js`
- Can cause 33.333% + 33.333% + 33.334% = 100.0000000001 → rejected incorrectly
- Or sum as 99.9999999999 → silently allows invalid state

**Fix:** Use Decimal.js for all financial calculations:
```typescript
const sum = allocations.reduce(
  (acc, a) => acc.plus(new Decimal(a.allocationPct)),
  new Decimal(0)
)
if (sum.gte(100)) throw new Error(...)
```

**Status:** ⚠️ IMPLEMENT THIS SPRINT

### [MEDIUM] Silent Failures in Error Handling

**Severity:** MEDIUM  
**Category:** Observability / Debugging  
**Related to:** Silent Failure Hunter findings (8 issues)

**Key areas:**
1. FX rate fetch: generic error logging without error ID → impossible to debug
2. Stella advisor: all errors mapped to "UNAUTHORIZED" → hides DB failures
3. Confidence score: console-only logging, no Sentry
4. Audit logging: no error handling for DB insert

**Impact:**
- Production debugging impossible
- Outages misdiagnosed as permission errors
- No visibility into systemic failures

**Recommended fix:**
1. Create `logError()` function that sends to Sentry (not just console)
2. Replace all broad `catch` blocks with specific exception handling
3. Add error context: org ID, user ID, operation name
4. Use error IDs (UUID) to link logs across systems

**Status:** ⚠️ IMPLEMENT THIS SPRINT

### [MEDIUM] Stella Quota Check Not Transactional

**Severity:** MEDIUM  
**Category:** Concurrency / Fairness  
**CWE:** CWE-362 (Concurrent Execution using Shared Resource)

**What it is:**
- Quota check and DB insert are NOT atomic
- Request at month boundary: passes check (July quota), inserts (August quota)
- Organization charged incorrectly

**Impact:**
- Quota enforcement inaccurate
- Billing disputes

**Fix:** Use database transaction:
```sql
BEGIN;
SELECT stella_monthly_quota ... FOR UPDATE;
SELECT COUNT(*) FROM stella_interactions WHERE organization_id = ? AND created_at BETWEEN month_start AND month_end;
IF count >= quota THEN ROLLBACK; ELSE INSERT; COMMIT; END IF;
```

**Status:** ⚠️ FIX THIS SPRINT

---

### [MEDIUM] No Timeout/Retry Logic on Gemini API Calls

**Severity:** MEDIUM  
**Category:** Resilience  
**CWE:** CWE-400 (Uncontrolled Resource Consumption)

**What it is:**
- Timeout: 30 seconds (reasonable)
- No retry logic with exponential backoff
- No circuit breaker pattern
- If Gemini is slow, request fails immediately

**Impact:**
- Transient failures fail permanently (should retry)
- No cascade protection if Gemini is down

**Fix:**
1. Add retry: up to 3 attempts with 1s, 2s, 4s delays
2. Circuit breaker: if >5 consecutive failures, reject new requests for 30s
3. Semaphore: limit concurrent Gemini requests to 5

**Status:** ⚠️ FIX THIS SPRINT

---

### [MEDIUM] Admin Audit Logging Lacks IP/User Agent

**Severity:** MEDIUM  
**Category:** Audit Trail Completeness  
**CWE:** CWE-778 (Insufficient Logging)

**What it is:**
- Admin actions ARE logged to audit_logs
- But NO IP address or user agent captured
- Rogue admin can modify production without accountability

**Fix:**
1. Capture `ipAddress` and `userAgent` from request headers
2. Flag high-risk actions (org suspension, quota = 0) with `risk_level: 'high'`
3. Alert on high-risk changes

**Status:** ⚠️ IMPLEMENT THIS SPRINT

## Good Practices (Maintain)

✅ **SROI Calculation Engine:**
- Deterministic (no randomness, reproducible)
- Precise (Decimal.js, 4-6 decimal places)
- Immutable (append-only triggers, insert-only lineItems)

✅ **Audit Trail:**
- Append-only by schema constraint
- Before/after JSON snapshots
- Comprehensive coverage

✅ **Pure Functions:**
- FX conversion (convertToUsd, convertFromUsd)
- Funder breakdown (computeFundersBreakdown)
- Allocation caps (wouldExceedCap)
- Well-tested in isolation

✅ **Auth & Permissions:**
- Role hierarchy centralized
- Permission checks pervasive
- SuperAdmin bypass explicit

✅ **Context Builders:**
- Stella context excludes file paths, raw content, PII (currently)
- Metadata-only approach is correct

✅ **Drizzle ORM Usage:**
- Parameterized queries prevent SQL injection
- Type-safe query builders
